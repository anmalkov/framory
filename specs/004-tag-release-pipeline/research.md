# Research: Tag-Triggered Release Pipeline

**Feature**: 004-tag-release-pipeline  
**Date**: 2026-03-03

## 1. Docker Image Semver Tagging

**Decision**: Use `docker/metadata-action@v5` with `type=semver` patterns to generate multi-level tags.

**Configuration**:
```yaml
- uses: docker/metadata-action@v5
  id: meta
  with:
    images: anmalkov/framory
    tags: |
      type=semver,pattern={{version}}
      type=semver,pattern={{major}}.{{minor}}
      type=semver,pattern={{major}}
```

For a `v1.2.3` tag push, this produces: `1.2.3`, `1.2`, `1`, `latest`.

**Rationale**: The action handles semver parsing, tag sanitization, and Docker registry compliance automatically. The `latest` tag is auto-generated by default for semver patterns. No shell scripting or regex needed.

**Alternatives considered**:
- Manual tag extraction via shell (`echo ${GITHUB_REF#refs/tags/v}`) — error-prone, no built-in sanitization
- Hardcoded tag list — inflexible, would require workflow changes for each release

## 2. Docker Build Caching

**Decision**: Use `type=gha,mode=max` cache with Docker Buildx for multi-stage builds.

**Configuration**:
```yaml
- uses: docker/setup-buildx-action@v3
- uses: docker/build-push-action@v6
  with:
    context: .
    push: true
    tags: ${{ steps.meta.outputs.tags }}
    labels: ${{ steps.meta.outputs.labels }}
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

**Rationale**: `mode=max` caches all layers including intermediate stages (Node.js frontend build + Python backend deps). GitHub Actions cache API is natively integrated — no external registry or manual cache management. Buildx is required and set up via `docker/setup-buildx-action`.

**Alternatives considered**:
- `type=inline` — only supports `mode=min`, caches only final stage layers (bad for multi-stage)
- `type=registry` — stores cache in Docker Hub, slower round-trips, consumes pull quota
- No caching — every build reinstalls npm and pip dependencies from scratch (~2-4 min wasted)

## 3. GitHub Release Notes Generation

**Decision**: Use the GitHub CLI (`gh release create --generate-notes`) which invokes native GitHub API auto-generated release notes.

**Configuration**:
```yaml
- name: Create GitHub Release
  run: |
    gh release create "${{ github.ref_name }}" \
      --repo "${{ github.repository }}" \
      --title "Release ${{ github.ref_name }}" \
      --generate-notes
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Rationale**: The GitHub CLI is pre-installed on `ubuntu-latest` runners. Native release notes use GitHub's database of commits and merged PRs between tags for accurate changelog generation. Supports optional `.github/release.yml` customization for categories. No third-party action dependency needed.

**Alternatives considered**:
- `softprops/action-gh-release@v2` — third-party community action, adds an external dependency that could fall behind GitHub API updates
- Manual `git log` changelog — loses PR context and contributor attribution
- Conventional Commits parsing — requires commit convention enforcement, overkill for this project

## 4. Tag Filtering for Strict Semver

**Decision**: Use broad glob `v*.*.*` in the trigger, combined with a job-level `if` condition that excludes pre-release tags (tags containing `-` or `+`).

**Configuration**:
```yaml
on:
  push:
    tags:
      - 'v*.*.*'

jobs:
  release:
    if: ${{ !contains(github.ref_name, '-') && !contains(github.ref_name, '+') }}
```

**Rationale**: GitHub Actions glob patterns are not regex — they cannot express "digits only" or negative patterns. The glob `v*.*.*` catches all semver-like tags, while the `if` condition rejects pre-release (`v1.0.0-rc1`) and build metadata (`v1.0.0+build.1`) tags. This is simple, readable, and covers all edge cases from the spec.

**Alternatives considered**:
- Shell regex validation step — adds a step that can fail, more complex
- Multiple glob patterns for digit combinations (`v[0-9].[0-9].[0-9]`, `v[0-9][0-9].[0-9].[0-9]`, etc.) — tedious and incomplete
- Triggering on all tags (`v*`) — too broad, would match `v-something`

## 5. Docker Hub Authentication

**Decision**: Use `docker/login-action@v3` with a Personal Access Token stored as a GitHub repository secret.

**Required secrets**:
| Secret | Value |
|--------|-------|
| `DOCKERHUB_USERNAME` | Docker Hub account username |
| `DOCKERHUB_TOKEN` | Docker Hub Personal Access Token (read/write scope) |

**Rationale**: Personal Access Tokens are Docker Hub's recommended authentication method (not account passwords). They support granular scopes and can be rotated independently. The login action handles credential management securely.

**Alternatives considered**:
- Account password — deprecated by Docker Hub, no scope granularity
- GitHub Container Registry (GHCR) — would require changing the spec's target registry, simpler auth but different from user's requirement

## 6. Preventing Partial Releases

**Decision**: Use sequential steps within a single job. GitHub Actions stops executing subsequent steps when any step fails (default behavior). No explicit `if` conditions or multi-job `needs` required for this simple workflow.

**Rationale**: For a linear pipeline (checkout → login → meta → build → release), step ordering within a single job is sufficient. If the Docker build/push fails, the release step is never reached. A single job is simpler than splitting into two jobs with `needs` — fewer YAML lines, no need to pass outputs between jobs, and the default failure behavior already prevents partial releases.

**Alternatives considered**:
- Multi-job workflow with `needs: build-and-push` — better separation of concerns but adds complexity for only 5 sequential steps
- Explicit `if: success()` on the release step — redundant since it's the default behavior, but could be added for readability
